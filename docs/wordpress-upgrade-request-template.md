# WordPress / PHP / Plugin Upgrade Request Template

Created: 2026-07-14 JST

別プロジェクトで、WordPress本体・PHP・プラグイン更新を安全に進めるための依頼プロンプト兼忘れがきです。

## Copy-Paste Prompt

以下を新しいCodexスレッドや作業依頼に貼り付けて使う。

```text
WordPressサイトの脆弱性確認と、WordPress本体・PHP・プラグインのバージョンアップ対応をお願いします。

方針:
- 本番反映はしないで、必ずテスト環境から進める
- テスト環境で問題ないことを確認してから、本番反映の可否を判断する
- 作業前に現状調査、バックアップ確認、差分記録を行う
- 可能ならGitHubリポジトリに、本番用ブランチとテスト用ブランチを作って作業記録を残す
- テンプレートやテーマ改修は必要最小限にし、既存サイトの見た目・挙動を維持する
- 一時メンテナンススクリプトを使う場合は、実行後に必ず削除する
- 更新後は公開ページでPHP Warning / Fatal error / Deprecated / Noticeが出ていないか確認する

対象サイト:
- 本番URL: 【本番URL】
- テストURL: 【テストURL】
- GitHubリポジトリ: 【GitHub URL】

接続情報:
- テストFTP/SFTP: 【ホスト / ID / パスワード / ルートパス】
- 本番FTP/SFTP: 【ホスト / ID / パスワード / ルートパス】
- WordPress管理画面: 【必要ならURL / ID / パスワード】
- DBバックアップ状況: 【取得済み / 未取得 / 取得方法】

進めたい作業:
1. 現状確認
   - PHPバージョン
   - WordPressバージョン
   - 有効プラグイン一覧と各バージョン
   - テーマ名とカスタマイズ有無
   - エラーログ
   - 公開ページの表示状態

2. 脆弱性・更新可否確認
   - WordPress本体の更新候補
   - PHPバージョン要件
   - 各プラグインの最新バージョンとPHP要件
   - 古い/配布停止/互換性不明のプラグイン
   - 本番だけに存在するプラグインがあれば、テスト環境にも反映して検証する

3. テスト環境で段階更新
   - まずPHPを本番と同じバージョンに合わせる
   - WordPress本体を更新
   - プラグインは一括ではなく、リスクの高いものから1つずつ更新して都度QA
   - つまずいたプラグインは無理に進めず、原因・代替案・ロールバック可否を整理する

4. Warning / Fatal / Deprecated対応
   - 公開画面に出るWarningを修正
   - error_logも確認
   - 既存テーマやプラグインの軽微なPHP互換修正が必要なら、バックアップを取って最小変更で対応する

5. QA
   - トップページ
   - 主要固定ページ
   - 投稿/ニュース一覧・詳細
   - 問い合わせフォーム
   - 画像スライダー
   - メニュー/フッター
   - スマホ幅/PC幅
   - REST APIやフォームAPIが必要なら確認

6. 記録
   - 更新前後のバージョン
   - 変更ファイル
   - バックアップパス
   - 実施したQA
   - 残リスク
   - 本番反映手順
   - ロールバック手順

本番反映は、テスト環境でこちらがOKを出すまで実施しないでください。
```

## Work Checklist

### 1. Before Touching Anything

- テスト環境と本番環境を取り違えない。
- 本番作業は禁止か、許可済みかを毎回確認する。
- DBバックアップ済みか確認する。
- ファイルバックアップの保存先を決める。
- FTP/SFTPの公開ルートを確認する。
- WordPressがサブディレクトリ設置か確認する。例: `/cms`
- 一時スクリプト名は推測されにくい名前にし、トークン必須にする。
- 実行後、一時スクリプトの削除確認まで行う。

### 2. Recommended Branches

- `main`: 作業記録・本番状態の記録
- `staging`: テスト環境向けの記録や差分
- `production`: 本番反映時の記録

実サイトがGit管理されていない場合でも、以下はGitHubに残す。

- 調査結果
- バージョン一覧
- 作業ログ
- QA結果
- ロールバック方法

### 3. Version Checks

確認項目:

- PHP CLIではなく、Web実行時のPHPバージョン
- WordPress本体バージョン
- DB更新が必要か
- 有効プラグイン
- 無効だが本番に残っている古いプラグイン
- テーマバージョン
- Must Use Plugin
- `wp-config.php` のデバッグ設定
- `error_log` / `php_error.log` / サーバー管理画面のログ

### 4. Update Order

基本順序:

1. テスト環境のバックアップ確認
2. テスト環境を本番に近い状態へ合わせる
3. PHPを予定バージョンへ変更
4. 表示確認
5. WordPress本体更新
6. DBアップグレード
7. 表示確認
8. プラグインを1つずつ更新
9. 各更新後に公開ページ確認
10. Warning/Fatal/Deprecated修正
11. フォーム・スライダー・ニュース・主要ページ確認
12. 本番反映手順を作成
13. ユーザー承認後に本番反映

### 5. QA Pages

最低限:

- `/`
- `/privacy/`
- `/contact/` または問い合わせページ
- ニュース一覧
- ニュース詳細
- 主要固定ページ
- 画像スライダーがあるページ
- フォーム送信画面
- `wp-json` を使っている場合はREST API

確認する文言:

- `Fatal error`
- `Parse error`
- `Warning:`
- `Notice:`
- `Deprecated:`
- `There has been a critical error`

### 6. Plugin Risk Notes

特に注意するもの:

- 配布停止プラグイン
- PHP要件が上がったプラグイン
- ショートコード系
- フォーム系
- スライダー系
- セキュリティ系
- キャッシュ/最適化系
- 古いテーマに依存するプラグイン

進め方:

- 高リスクプラグインは一括更新しない。
- 更新前にディレクトリ単位でバックアップする。
- 更新後すぐに公開ページとログを見る。
- 壊れたら、更新前ディレクトリに戻せる状態を保つ。

### 7. Production Deployment Rules

- テスト環境でユーザー確認が終わるまで本番は触らない。
- 本番作業前に、作業対象を再確認する。
- 本番DBバックアップ済みか確認する。
- 本番ファイルバックアップを取得する。
- 本番反映はテストと同じ順番で行う。
- 本番反映後はテスト時と同じQAを実施する。
- 本番で不具合が出たら、更新前バックアップへ戻す。

### 8. Final Report Format

```text
対応完了しました。

対象:
- テスト環境: 【URL】
- 本番環境: 【未反映 / 反映済み】

更新後:
- PHP: 【version】
- WordPress: 【version】
- 更新プラグイン:
  - 【plugin】: 【before】 -> 【after】

実施内容:
- 【内容】

QA:
- 【URL】: 200 OK / Warningなし
- 【URL】: 200 OK / Warningなし

バックアップ:
- DB: 【場所/日時】
- Files: 【場所/日時】

残リスク:
- 【あれば】

次の判断:
- 本番反映OKなら、同じ手順で本番に進めます。
```

## Lessons From Rikusan Work

- 本番反映前に、必ず「本番はまだ触らない」かを確認する。
- 本番にだけある古いプラグインは、テストにも入れてから検証する。
- PHPバージョンはCLIではなくWeb表示で確認する。
- Autoptimizeなどのキャッシュがある場合、CSS更新後にキャッシュクリアが必要。
- テーマのハードコードフッターはWordPressメニューでは直せないことがある。
- ページ本文にCSSを入れる場合、WordPressの自動整形で`style`内に`p`タグが入らないよう1行化する。
- 作業スクリプトは実行後に削除し、FTPで存在しないことまで確認する。
